## 2. Action Plan

### Fix 1: Robust JSON Serialization
**Problem:** The current logging mechanism fails when encountering circular references or BigInt values in objects.
**Solution:**
- Implement `safeJsonStringify` helper.
- Use `replacer` function to handle BigInt and circular references.
- Apply this helper in all logging and payload serialization points.

### Fix 2: SQLite Concurrency Management
**Problem:** High-concurrency scenarios cause `SQLITE_BUSY` errors due to default locking behavior.
**Solution:**
- Implement a `sqlite-retry` wrapper logic using exponential backoff.
- Configure SQLite `busy_timeout` to a higher value (e.g., 5000ms).
- Ensure transactions are short-lived.

### Fix 3: Zombie Job Cleanup (Startup Recovery)
**Problem:** If the worker crashes while a job is `PROCESSING`, the job remains locked forever upon restart.
**Solution:**
- Create a `cleanupStaleJobs()` function to run at application startup.
- Query for jobs in `PROCESSING` state.
- Reset them to `QUEUED` (for retry) or `FAILED` (if max retries exceeded or for safety).
- Log the number of recovered jobs.

### Fix 4: Dual-Write Consistency (Compensation Transaction)
**Problem:** Writing to SQLite and then pushing to Redis is not atomic. If Redis fails, the DB record exists but will never be processed.
**Solution:**
- Wrap the operation in a try/catch block.
- **Try:** Insert into SQLite -> Push to Redis.
- **Catch:** If Redis push fails, delete the record from SQLite (Compensation).
- **Finally:** Throw error to caller so API/CLI knows the request failed.
