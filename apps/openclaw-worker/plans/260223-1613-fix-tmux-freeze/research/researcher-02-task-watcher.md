# Research Report: Tmux Session Initialization & Rate Limiting

## Findings

1. **Initialization Location**: The core tmux session creation logic resides in `lib/brain-process-manager.js` inside the `spawnBrain()` function. It creates the session `tom_hum_brain` and executes the loop for `--parallel` capability (splitting windows into multiple panes).
2. **Task Watcher Invocation**: `task-watcher.js` imports and invokes `spawnBrain()` wrapped in a `safeBoot` function block.
3. **Current Rate Limiting Status**: Reviewing `task-watcher.js` reveals that the rate-limiting fix *is actually already present* (lines 188-200). It applies the limits immediately after spawning the brain:
   ```javascript
   execSync('tmux set-option -g c0-change-interval 300 2>/dev/null || true');
   execSync('tmux set-option -g c0-change-trigger 50 2>/dev/null || true');
   ```

## Impact on `--parallel` Capability
Because the `c0-change-*` options are applied globally (`-g`) or session-wide, they automatically govern all panes generated by the parallel pane-splitting loop in `spawnBrain()`. Therefore, the `--parallel` capability remains **completely intact**.

## Recommendations for Refinement
- **Session Isolation**: Using `-g` applies these limits to *all* tmux sessions on the host machine. To avoid negatively affecting the user's personal terminal sessions, it is recommended to apply these settings specifically to the target session using `-t tom_hum_brain` instead of `-g`.
- **Code Encapsulation**: Consider moving these rate-limiting commands directly into `lib/brain-process-manager.js` right after `tmux new-session`. This ensures all tmux configuration logic is centralized in one place.
