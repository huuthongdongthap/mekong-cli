MISSION: Implement a Production-Ready Account & Ghost Protocol Manager to prepare for live trading with real funds. This module will handle secure storage of exchange API credentials, manage a pool of residential proxies (BrightData, Oxylabs), configure and fine-tune all anti-detection parameters (jitter, fingerprint rotation, WebSocket sharding), and continuously monitor for any anti-bot flags or rate-limiting from exchanges. It must integrate seamlessly with the existing Phase 6 Ghost Protocol modules and provide a safe, auditable way to transition from simulation to live trading. All code must be in TypeScript with 100% Jest test coverage (using mocks for external proxy services and exchange APIs). Provide a configuration file `config.live.json`.

**MODULE: Live Trading Orchestrator & Ghost Protocol Manager**
- File path: `apps/algo-trader/src/live/`
- Subcomponents:
  - `accountVault.ts`: Securely stores and retrieves exchange API credentials using environment variables or an encrypted local file (with a master password). Supports multiple accounts per exchange (e.g., VIP1, VIP2). Implements key rotation and expiry checks.
    - Uses `crypto` module for AES-256 encryption.
    - Credentials are never logged or exposed in error messages.
  - `proxyPoolManager.ts`: Integrates with residential proxy providers (BrightData, Oxylabs) via their APIs to fetch a dynamic list of proxies. Features:
    - Automatic rotation every N seconds (configurable).
    - Proxy health checks (latency, success rate).
    - Fallback to direct connection if pool is exhausted.
    - Support for sticky sessions (same proxy for a session) if required by exchange.
  - `ghostConfigurator.ts`: Reads the base Ghost Protocol configuration (from Phase 6) and applies live-specific overrides:
    - Jitter parameters (mean, std) based on recent measurements of human reaction time (can be set manually or via auto-tuning from Prompt 5).
    - Fingerprint selection strategy (rotate per request, per session, etc.).
    - WebSocket shard count and distribution logic (from Phase 6).
    - Enables or disables certain noise actions (Phase 6 Chameleon) based on exchange risk profile.
  - `antiBotMonitor.ts`: Continuously monitors exchange responses for signs of anti-bot detection:
    - HTTP status codes (429 rate limit, 403 forbidden).
    - Custom error messages (e.g., "request rejected", "too many requests").
    - Sudden increase in slippage or order rejections.
    - Unusual latency patterns.
    - When suspicious activity is detected, it can:
      - Rotate proxy immediately.
      - Increase jitter.
      - Switch to a different account.
      - Pause trading on that exchange and alert.
  - `complianceLogger.ts`: Maintains an immutable log of all live trading activities for audit and tax purposes. Logs include:
    - Timestamp, exchange, symbol, side, quantity, price, fee.
    - Proxy used, fingerprint ID.
    - Any anti-bot events.
    - Logs are encrypted and stored locally with optional cloud backup.
  - `canaryPreparer.ts`: Prepares the system for canary deployment by:
    - Creating a new isolated instance (e.g., Docker container) with live configuration.
    - Performing a series of connectivity and authentication tests (dry-run orders with minimal quantity, then cancel).
    - Validating that Ghost Protocol is functioning (no rate limits, correct IP geolocation, etc.).
    - Generating a readiness report.
  - `index.ts`: Main entry point that initializes all components, starts monitoring, and exposes a control interface (HTTP or CLI) for manual intervention.

**Implementation Notes**:
- For proxy integration, use `axios` with proxy support. The proxy list should be fetched and cached, with background refresh.
- The account vault must never write secrets to disk unencrypted. Use `dotenv` for environment variables during development, but for production, recommend using a secret management service (HashiCorp Vault, AWS Secrets Manager) – provide hooks.
- The anti-bot monitor should use statistical process control (e.g., moving averages) to detect anomalies.
- Compliance logs should be in a format easily importable into accounting software (CSV with specific columns). Provide a conversion script.
- All components must respect the `canaryMode` flag: if true, orders are either simulated or sent with minimal quantity (e.g., 1 USDT) and immediately canceled.

**Testing & Quality**:
- Achieve 100% Jest coverage for all TypeScript files. Use `jest.mock` for proxy provider APIs and exchange clients.
- Test account vault with encrypted storage and key rotation.
- Simulate anti-bot events and verify monitor reactions.
- Ensure that in canary mode, no real orders are placed (or they are automatically canceled).
