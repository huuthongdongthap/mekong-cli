MISSION: Implement an Advanced Anti-Bot & Rate Limit Monitoring System (AntiBotSentinel) that continuously observes all exchange interactions in real time, detects patterns indicative of anti-bot detection (rate limiting, order rejection, toxic flow flags), and automatically triggers defensive actions (proxy rotation, jitter adjustment, account switching, or pausing trading on specific symbols). This module builds on the Ghost Protocol (Phase 6) and integrates with the Live Trading Orchestrator (Prompt 6) and Canary Deployment (Prompt 7). It must provide a live dashboard view of exchange health and any defensive actions taken. All code must be in TypeScript with 100% Jest test coverage (using mocks for exchange APIs and time-series data). Provide a configuration file `config.antibot.json`.

**MODULE: AntiBotSentinel**
- File path: `apps/algo-trader/src/monitoring/antiBot/`
- Subcomponents:
  - `dataCollector.ts`: Subscribes to all exchange API calls and WebSocket messages (via hooks in the exchange connector). Captures:
    - HTTP status codes, response times, error messages.
    - Order placement results (filled, rejected, partially filled).
    - WebSocket disconnections or unusual gaps.
    - Slippage actual vs. expected.
    - Any exchange-specific anti-bot headers or response flags.
    - Stores events in an in-memory time-series buffer (e.g., last 1000 events per exchange) and optionally writes to a persistent log for post-mortem analysis.
  - `patternDetector.ts`: Analyzes the collected data using statistical and ML techniques to detect anomalies:
    - **Rate limit detection**: counts requests per second per endpoint, compares to known exchange limits. Triggers if approaching limit.
    - **Error spike detection**: moving average of 4xx/5xx errors, triggers if > threshold.
    - **Order rejection analysis**: clusters rejection reasons (e.g., "-1021: Timestamp for this request is outside of the recvWindow") and flags if frequency increases.
    - **Slippage anomaly**: compares actual slippage to expected (from orderbook depth). If slippage consistently exceeds a multiplier, signals possible toxic flow.
    - **Latency shift**: monitors average response latency; sudden increase may indicate network throttling.
    - **Machine learning option**: optional lightweight online model (Isolation Forest or simple threshold) trained on historical good behavior to flag outliers.
  - `responseEngine.ts`: Maps detected patterns to defensive actions:
    - **Proxy rotation**: force immediate proxy change (via Phase 6 proxy manager).
    - **Jitter adjustment**: temporarily increase jitter mean or std.
    - **WebSocket shard rebalance**: reconnect shards with new fingerprints.
    - **Account switch**: if multiple VIP accounts configured, switch to a different account.
    - **Symbol pause**: stop trading the affected symbol(s) for a cooldown period.
    - **Global pause**: emergency halt all trading on that exchange.
    - **Alert**: send notification via Telegram/Slack (reuse alert manager from Prompt 2).
    Actions are configurable per exchange and per severity level.
  - `coordinator.ts`: Main loop that:
    - Periodically (e.g., every second) triggers pattern detection on recent data.
    - If a pattern is confirmed, calls `responseEngine` with appropriate severity.
    - Logs all detections and actions to a dedicated `antiBot.log`.
    - Exposes a real-time metrics feed via WebSocket for dashboard.
  - `dashboardIntegration.ts`: Adds new panels to the NextJS dashboard:
    - Exchange health score (0–100) per exchange.
    - Recent events timeline (rate limits, rejections, actions).
    - Current active defenses (e.g., "proxy rotated 2 min ago", "jitter increased").
    - Manual override buttons to trigger actions (rotate proxy, pause symbol, etc.).

**Implementation Notes**:

- All actions must be idempotent and respect cooldown periods to avoid flapping.
- The dashboard should use the existing WebSocket infrastructure; add new message types and update the React components accordingly.
