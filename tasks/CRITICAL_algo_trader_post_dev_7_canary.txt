MISSION: Implement a Canary Deployment System that safely introduces live trading with real funds in incremental steps (starting at 1% of capital). This module builds upon the Live Trading Orchestrator (Prompt 6/9) and the Chaos Testing framework (Prompt 2/9) to create a controlled, observable rollout process. It will manage two parallel instances: the current baseline (if any) and the canary instance running the latest code/configuration. It continuously compares key metrics (Sharpe, slippage, error rates, anti-bot flags) and automatically rolls back if performance degrades. All code must be in TypeScript with 100% Jest test coverage (using mocks for exchange APIs and metrics). Provide a configuration file `config.canary.json` and integrate with the existing NextJS dashboard.

**MODULE: Canary Deployment Controller**
- File path: `apps/algo-trader/src/deployment/canary/`
- Subcomponents:
  - `instanceManager.ts`: Manages two runtime instances:
    - **Baseline**: the current stable version (could be a previous release, or a mock that just records metrics).
    - **Canary**: the new version with the latest code and configuration.
    - Uses Docker (via `dockerode`) or process forks to run isolated instances. For simulation, can run in-memory with separate event loops.
    - Ensures that both instances receive the same market data feed (e.g., by duplicating WebSocket streams).
  - `trafficSplitter.ts`: Routes a configurable percentage of real capital (starting at 1%) to the canary instance. The rest goes to the baseline. Supports:
    - Percentage-based split (e.g., 1% of orders go to canary).
    - Symbol-based split (certain symbols traded only by canary).
    - Time-based split (canary trades during specific hours).
  - `metricsComparator.ts`: Collects real-time metrics from both instances (via their WebSocket metrics feeds) and computes:
    - Sharpe ratio (rolling window, e.g., 1 hour).
    - Average slippage in basis points.
    - Order fill rate / rejection rate.
    - API error rate (5xx, rate limits).
    - Anti-bot flags (from Phase 6 monitor).
    - Latency percentiles.
    - Performs statistical tests (t-test or Mann-Whitney) to determine if canary significantly underperforms.
  - `rollbackTrigger.ts`: If any metric exceeds a threshold (configurable) or the statistical test shows degradation with confidence >95%, automatically:
    - Sets canary traffic to 0%.
    - Logs the incident with all relevant data.
    - Sends alerts via Telegram/Slack (integrate with `alertManager` from Prompt 2).
    - Optionally restarts the canary with previous configuration for debugging.
  - `promotionManager.ts`: After a successful evaluation period (e.g., 24 hours with no issues), automatically promotes the canary to baseline:
    - Updates the main configuration to use the new version.
    - Shuts down the old baseline.
    - Sends a notification.
  - `dashboardIntegration.ts`: Exposes a new WebSocket feed with canary status:
    - Current traffic split.
    - Live metric comparison charts.
    - Rollback history.
    - Manual override controls (set split, force rollback, force promotion).
  - `index.ts`: Main entry point that reads config, starts the instance manager, and orchestrates the canary lifecycle.

**Integration & Configuration**:
- Create `config.canary.json`:
  ```json
  {
    "baseline": {
      "image": "algo-trader:stable",
      "configPath": "./config.stable.json"
    },
    "canary": {
      "image": "algo-trader:latest",
      "configPath": "./config.canary.json"
    },
    "trafficSplit": {
      "initialPercent": 1,
      "incrementSteps": [5, 10, 25, 50, 100],
      "evaluationPeriodHours": 24,
      "symbols": ["BTC/USDT", "ETH/USDT"]  // if empty, all symbols
    },
    "metrics": {
      "sharpeWindowHours": 1,
      "slippageThresholdBps": 5,
      "errorRateThreshold": 0.01,
      "antiBotFlagThreshold": 1,
      "confidenceLevel": 0.95
    },
    "alerts": {
      "telegramBotTokenEnv": "TELEGRAM_BOT_TOKEN",
      "telegramChatIdEnv": "TELEGRAM_CHAT_ID",
      "slackWebhookEnv": "SLACK_WEBHOOK"
    },
    "dashboardPort": 3001
  }
  ```
- Provide npm scripts:
  - `npm run canary:start` – starts the canary deployment controller.
  - `npm run canary:status` – shows current status.
  - `npm run canary:promote` – manually promote canary.
  - `npm run canary:rollback` – manually rollback.
- The controller should run as a separate process (or container) that supervises the two trading instances.

**Implementation Notes**:
- Use `dockerode` to manage containers if using Docker. For testing, use in-memory child processes with `child_process.fork()` and IPC for metrics.
- The metrics comparator should use streaming algorithms (e.g., Welford's online algorithm for variance) to avoid storing all data points.
- For statistical testing, use `jstat` or a simple t-test implementation.
- Ensure that the traffic splitter works at the order level: when the core engine wants to send an order, it checks a deterministic hash (e.g., of the symbol + timestamp) to decide which instance executes it. This requires modifying the exchange connector to support dual routing – or the controller can act as a proxy.
- All alerts must include a link to the dashboard for immediate investigation.
- The dashboard should have manual override buttons (protected by a simple API key).

**Testing & Quality**:

- Simulate metric degradation and verify automatic rollback.
- Test promotion flow and ensure baseline is safely shut down.
